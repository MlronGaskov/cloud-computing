# Cloud Computing on Clojure

Прототип облачной вычислительной платформы, позволяющей вызывать функции на удалённых узлах кластера так же, как локальные функции.

Кластер состоит из **координатора** и набора **воркеров**: координатор принимает регистрацию функций и задач, выбирает свободные узлы и распределяет нагрузку, а воркеры получают код и данные, загружают переданные зависимости и выполняют вычисления.

## Что реализовано

### 1. Базовый механизм передачи и запуска кода и данных на Clojure, а также балансировка нагрузки
- Реализован кластер из **coordinator** и **worker**-узлов.
- Пользовательский код передаётся на удалённый узел в виде **bundle**: вместе с точкой входа отправляются исходники всех необходимых `namespace`, включая транзитивные зависимости.
- Аргументы и результаты вызова передаются по сети с помощью сериализации.
- Координатор хранит список зарегистрированных функций и воркеров, отслеживает занятость узлов и направляет задачу на свободный воркер.
- Если свободного воркера нет, задача помещается в очередь и запускается позже.

### 2. Макросы для декларации удалённого вызова и распределённого `map`
- Реализован макрос **`defremote`**, который объявляет обычную удалённо исполняемую функцию.
- Для прикладного кода вызов такой функции выглядит как обычный вызов Clojure-функции.
- Реализован отдельный макрос **`defdmap`**, который объявляет распределённый `map` для одноаргументной функции.
- Таким образом, обычный удалённый вызов и распределённая пакетная обработка разделены явно:
    - `defremote` — удалённый вызов одной задачи;
    - `defdmap` — распределённая обработка коллекции задач.
- При регистрации удалённой функции система автоматически строит bundle с кодом и зависимостями и отправляет его координатору.

### 3. Распределённый и параллельный `map` с учётом многоядерности
- Для одноаргументных функций реализован отдельный режим **distributed map** через `defdmap`.
- При вызове такой функции система:
    - принимает входную коллекцию,
    - разбивает её на чанки,
    - отправляет чанки как отдельные удалённые batch-задачи,
    - распределяет их между воркерами,
    - внутри воркера обрабатывает элементы чанка локально в несколько потоков.
- Параметр `:workers` задаёт, сколько batch-вызовов может одновременно находиться “в полёте”.
- Параметр `:threads` задаёт локальный уровень параллелизма внутри одного воркера при обработке элементов чанка.

### 4. Поддержка транспорта Java-объектов, Java-лямбд и элементов Java Stream API
- Поддержана передача **пользовательских Java DTO**.
- Поддержана передача **обычных serializable Java-объектов** из стандартной библиотеки.
- Поддержана передача **serializable Java functional objects**.
- Поддержана передача **serializable Java lambdas**.
- Поддержана передача и выполнение вычислений, описанных через **Java Stream API**.

## Архитектура

Система состоит из трёх основных частей:

- **Client** - прикладной код, который объявляет и вызывает удалённые функции.
- **Coordinator** - центральный узел, который:
    - регистрирует воркеров,
    - хранит зарегистрированные функции,
    - принимает запросы на выполнение,
    - выбирает свободный воркер,
    - управляет очередью задач.
- **Worker** - удалённый исполнитель, который:
    - принимает bundle с кодом и аргументами,
    - загружает переданные namespace,
    - выполняет функцию,
    - возвращает результат.

## Кластер

Кластер состоит из **координатора (coordinator)** и набора **исполнителей (workers)**.

**Coordinator** - центральный узел системы. Он:
- принимает регистрацию воркеров;
- принимает запросы на выполнение;
- выбирает свободный воркер;
- ставит задачу в очередь, если все воркеры заняты;
- возвращает результат клиенту.

**Worker** - узел, который непосредственно выполняет вычисления. Он:
- регистрируется у координатора при старте;
- периодически отправляет heartbeat, чтобы координатор знал, что узел доступен;
- получает от координатора код функции и аргументы;
- загружает переданные зависимости;
- выполняет функцию и отправляет результат обратно.

Работа кластера выглядит так:

1. **Запускается coordinator** и начинает принимать HTTP-запросы.
2. **Запускаются workers**, каждый из них регистрируется у coordinator и начинает отправлять heartbeat.
3. **Клиентское приложение** подключается к coordinator и регистрирует удалённые функции.
4. При вызове удалённой функции клиент отправляет на coordinator:
    - идентификатор функции,
    - аргументы вызова.
5. Coordinator:
    - находит bundle с кодом функции,
    - выбирает свободный worker,
    - отправляет ему задачу на выполнение.
6. Worker:
    - принимает bundle и аргументы,
    - загружает переданный код,
    - вызывает entry-функцию,
    - возвращает результат.
7. Coordinator получает ответ от worker и передаёт его клиенту.

Если в момент запроса свободных worker нет, coordinator не теряет задачу, а помещает её в очередь. Как только один из воркеров освобождается, задача отправляется ему на выполнение.

Таким образом:
- **client** отвечает за объявление и вызов удалённых функций;
- **coordinator** отвечает за диспетчеризацию и балансировку нагрузки;
- **worker** отвечает за фактическое выполнение кода.

Такая схема позволяет отделить управление кластером от вычислений и распределять задачи между несколькими машинами без предварительного развёртывания пользовательского приложения на каждом узле.

## Передача кода и зависимостей

Система передаёт на удалённый узел не всё приложение пользователя целиком, а только тот набор namespace, который нужен для выполнения конкретной удалённой функции.

При регистрации удалённой функции система:
- определяет её точку входа;
- анализирует зависимости из формы `(ns ...)`;
- рекурсивно собирает все транзитивно зависимые `namespace`;
- формирует bundle, содержащий:
    - entry-функцию,
    - исходники всех нужных `namespace`,
    - порядок их загрузки.

Таким образом, система передаёт не весь проект, но и не минимальный набор отдельных `defn`: она передаёт только нужные namespace целиком.

Зависимости между namespace определяются по объявлениям в `ns`:
- `:require`
- `:use`
- `:require-macros`


## Как устроен транспорт Java-объектов

Помимо Clojure-данных система умеет передавать произвольные Java-объекты, если они сериализуемы.

При сериализации payload система:
1. рекурсивно обходит объектный граф передаваемого значения;
2. находит классы объектов, которые не относятся к простым примитивным значениям;
3. определяет, из какого `.jar` или директории загружен каждый класс;
4. добавляет соответствующие артефакты в транспортный пакет;
5. сериализует сам payload;
6. на принимающей стороне восстанавливает временный classloader из переданных артефактов;
7. десериализует payload уже в этом classloader.

Это позволяет передавать не только стандартные сериализуемые Java-объекты, но и пользовательские классы, даже если они заранее не установлены на worker.

Поддерживаются, например:
- пользовательские DTO из `src-java/demo/dto`;
- обычные объекты стандартной библиотеки Java, такие как `UUID`, `Date`, `ArrayList`, `HashMap`;
- serializable реализации Java functional interfaces.

## Как устроен транспорт Java-лямбд и functional objects

Java-лямбды и functional objects передаются не как исходный код, а как обычные сериализуемые Java-объекты.

Это работает так:
- если Java lambda реализует serializable functional interface, она может быть сериализована как объект;
- при обходе payload система видит такой объект, определяет его класс и подхватывает соответствующий `.jar` или директорию классов;
- на worker этот класс становится доступен через временный classloader;
- после этого лямбда или functional object может быть использована как `Function`, `Predicate`, `BinaryOperator` и т.д.

В проекте для этого используются:
- serializable functional interfaces (`SFunction`, `SPredicate`, `SBinaryOperator`);
- serializable lambda-фабрики в `demo.fn.Lambdas`;
- обычные class-based реализации `Function` / `Predicate` / `BinaryOperator`.

Важно: поддерживаются именно **Java serializable lambdas** и **serializable Java functional objects**.

## Как устроен транспорт и выполнение Java Stream API

В проекте поддерживается не передача “живого” объекта `Stream`, а передача плана вычисления.

Такой план включает:
- источник данных;
- последовательность операций, например `map` и `filter`;
- терминальную операцию, например `reduce` или `collect-to-list`.

На удалённой стороне worker:
- получает этот план как часть аргументов;
- восстанавливает все переданные Java-классы;
- в `cloud.java.stream/execute` заново строит stream из source;
- последовательно применяет операции;
- выполняет терминальную операцию и возвращает результат.

Внутри одного stream plan параллелизма нет: один plan выполняется последовательно. Параллелизм возможен только на уровне нескольких plan-ов / нескольких задач.


## Использование
### `defremote` 
- используется для создания одной задачи на координаторе
- принимает любые аргументы(не только коллекции) и функцию для этих аргументов
- получает один результат 
#### Пример
```clojure
(ns my.app
  (:require [cloud.remote :as remote]))

(remote/defremote remote-square [x]
  (* x x))

(defn -main [& _]
  (remote/connect! {:host "localhost" :port 8080})
  (remote/deploy!)

  (println (remote-square 7))) ; => 49
```

### `defdmap`
- принимает коллекцию, настройки распределенного запуска `{:workers W :threads T}`и функцию для одного элемента коллекции
- разбивает коллекцию на чанки, каджый чанк - одна задача для координатора
- внутри worker элементы чанка могут считаться параллельно в пуле потоков размера `:threads`
- результаты собираются обратно в один вектор в исходном порядке

#### Пример
```clojure
(ns my.app
  (:require [cloud.remote :as remote]))

(defn heavy [n]
  (reduce + (map #(* % %) (range n))))

(remote/defdmap remote-heavy-map {:workers 2 :threads 8} [n]
  (heavy n))

(defn -main [& _]
  (remote/connect! {:host "localhost" :port 8080})
  (remote/deploy!)

  (println (remote-heavy-map [1000 1200 1400 1600])))
