# Cloud Computing on Clojure

Прототип облачной вычислительной платформы, позволяющей вызывать функции на удалённых узлах кластера так же, как локальные функции.

Кластер состоит из **координатора** и набора **воркеров**: координатор принимает регистрацию функций и задач, выбирает свободные узлы и распределяет нагрузку, а воркеры получают код и данные, загружают переданные зависимости и выполняют вычисления.

## Что реализовано

### 1. Базовый механизм передачи и запуска кода и данных на Clojure, а также балансировка нагрузки
- Реализован кластер из **coordinator** и **worker**-узлов.
- Пользовательский код передаётся на удалённый узел в виде **bundle**: вместе с точкой входа отправляются исходники всех необходимых `namespace`, включая транзитивные зависимости.
- Аргументы и результаты вызова передаются по сети с помощью сериализации.
- Координатор хранит список зарегистрированных функций и воркеров, отслеживает занятость узлов и направляет задачу на свободный воркер.
- Если свободного воркера нет, задача помещается в очередь и запускается позже.

### 2. Макрос для декларации удалённо исполняемой функции
- Реализован макрос **`defremote`**, который позволяет объявить функцию как удалённую.
- Для прикладного кода вызов такой функции выглядит как обычный вызов Clojure-функции.
- При регистрации удалённой функции система автоматически строит bundle с кодом и зависимостями и отправляет его координатору.

### 3. Распределённый и параллельный `map` с учётом многоядерности
- Для одноаргументных удалённых функций реализован режим **distributed map**.
- Если такой функции передать коллекцию, система автоматически:
    - разбивает данные на части (**chunks**),
    - распределяет части по нескольким воркерам,
    - внутри каждого воркера обрабатывает элементы чанка в несколько потоков.

### 4. Поддержка транспорта Java-объектов, Java-лямбд и элементов Java Stream API
- Поддержана передача **пользовательских Java DTO**.
- Поддержана передача **serializable Java functional objects** и **serializable Java lambdas**.
- Поддержана передача и выполнение вычислений для **Java Stream API**.

## Архитектура

Система состоит из трёх основных частей:

- **Client** — прикладной код, который объявляет и вызывает удалённые функции.
- **Coordinator** — центральный узел, который:
    - регистрирует воркеров,
    - хранит зарегистрированные функции,
    - принимает запросы на выполнение,
    - выбирает свободный воркер,
    - управляет очередью задач.
- **Worker** — удалённый исполнитель, который:
    - принимает bundle с кодом и аргументами,
    - загружает переданные namespace,
    - выполняет функцию,
    - возвращает результат.

В первом терминале:

```bash
docker compose -f docker/docker-compose.yml up --build
```

Во втором терминале:

```bash
clojure -M:demo
```

## Кластер

Кластер состоит из **координатора (coordinator)** и набора **исполнителей (workers)**.

**Coordinator** — центральный узел системы. Он:
- принимает регистрацию воркеров;
- принимает запросы на выполнение;
- выбирает свободный воркер;
- ставит задачу в очередь, если все воркеры заняты;
- возвращает результат клиенту.

**Worker** — узел, который непосредственно выполняет вычисления. Он:
- регистрируется у координатора при старте;
- периодически отправляет heartbeat, чтобы координатор знал, что узел доступен;
- получает от координатора код функции и аргументы;
- загружает переданные зависимости;
- выполняет функцию и отправляет результат обратно.

### Взаимодействие узлов

Работа кластера выглядит так:

1. **Запускается coordinator** и начинает принимать HTTP-запросы.
2. **Запускаются workers**, каждый из них регистрируется у coordinator и начинает отправлять heartbeat.
3. **Клиентское приложение** подключается к coordinator и регистрирует удалённые функции.
4. При вызове удалённой функции клиент отправляет на coordinator:
    - идентификатор функции,
    - аргументы вызова.
5. Coordinator:
    - находит bundle с кодом функции,
    - выбирает свободный worker,
    - отправляет ему задачу на выполнение.
6. Worker:
    - принимает bundle и аргументы,
    - загружает переданный код,
    - вызывает entry-функцию,
    - возвращает результат.
7. Coordinator получает ответ от worker и передаёт его клиенту.

Если в момент запроса свободных worker нет, coordinator не теряет задачу, а помещает её в очередь. Как только один из воркеров освобождается, задача отправляется ему на выполнение.

### Роли компонентов

Таким образом:
- **client** отвечает за объявление и вызов удалённых функций;
- **coordinator** отвечает за диспетчеризацию и балансировку нагрузки;
- **worker** отвечает за фактическое выполнение кода.

Такая схема позволяет отделить управление кластером от вычислений и распределять задачи между несколькими машинами без предварительного развёртывания пользовательского приложения на каждом узле.

## Передача кода и зависимостей

Система передаёт на удалённый узел не всё приложение пользователя целиком, а только тот набор namespace, который нужен для выполнения конкретной удалённой функции.

### Что передаётся

При регистрации удалённой функции система:
- определяет её точку входа;
- анализирует зависимости из формы `(ns ...)`;
- рекурсивно собирает все транзитивно зависимые `namespace`;
- формирует bundle, содержащий:
    - entry-функцию,
    - исходники всех нужных `namespace`,
    - порядок их загрузки.

Таким образом, система передаёт не весь проект, но и не минимальный набор отдельных `defn`:  
она передаёт только нужные namespace целиком.

### Как находятся зависимости

Зависимости между namespace определяются по объявлениям в `ns`:
- `:require`
- `:use`
- `:require-macros`
